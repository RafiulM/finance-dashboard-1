# Finance Dashboard App Flow Document

## Onboarding and Sign-In/Sign-Up
A new visitor arrives on the Finance Dashboard home page via the root URL. On that landing page, the options to sign in or sign up are clearly presented. When choosing to create a new account, the user clicks the sign-up button and is taken to the `/sign-up` page. There they enter an email address and a password and submit the form. The application sends this data to the backend at `/api/auth` to create the account. Upon successful registration, the server issues a JSON Web Token in an HTTP-only cookie and automatically redirects the user to the main dashboard.

If an existing user visits the app, they click the sign-in link which navigates to the `/sign-in` page. They enter their credentials and submit. The front end makes a request to the same `/api/auth` endpoint, which validates the credentials, issues a JWT cookie, and redirects back to the dashboard. Signing out is accomplished by clicking the logout button in the header of any dashboard page, which clears the authentication cookie and returns the user to the `/sign-in` page.

## Main Dashboard or Home Page
After successful authentication, the user lands on the `/dashboard` route, which serves as the main dashboard view. The page is wrapped by a dashboard-specific layout that shows a vertical sidebar on the left and a header across the top. The sidebar contains navigation back to the dashboard and any future pages. The header displays the user’s email or name and a logout button on the right. The central content area shows summary cards that highlight total income, total expenses, and other key metrics, followed by chart components—such as bar or line charts—and a table of individual transactions drawn from the local data source.

The global layout defined at the root level ensures that headers, footers, and any shared navigation remain consistent across both the authentication and dashboard pages. The dashboard page itself uses a secondary layout to arrange components, handle padding, and maintain responsive behavior on mobile, tablet, and desktop screens.

## Detailed Feature Flows and Page Transitions
When the dashboard first loads, the front end fetches data from the static `data.json` file or, in later phases, from an external API. While data is loading, a spinner or placeholder is shown. Once the data arrives, the summary cards render followed by charts and the transaction table. If a user clicks on any future drill-down link—such as expanding a chart or viewing transaction details—they would navigate to a new route under `/dashboard`, all handled by the file-based routing system of Next.js.

If an unauthenticated user tries to access `/dashboard` directly, the application checks the presence of the JWT cookie on page load. If the token is missing or invalid, the user is redirected back to `/sign-in`. From any dashboard view, clicking logout instantly discards the cookie and navigates back to the login screen. Navigating between dashboard pages and components is seamless thanks to client-side routing, which prefetches code and data to keep transitions smooth.

## Settings and Account Management
In this version, account management is limited to logging out. The user’s email or name is displayed in the header for identification, but there is no profile edit or password reset page yet. Should those pages be added in the future, they will follow the same pattern: a dedicated route under `/dashboard` for profile settings, a form that sends updates to an API endpoint, and then a redirect back to the dashboard. After any settings change, the user is brought back to the main dashboard, preserving their context.

## Error States and Alternate Paths
If a user enters invalid credentials during sign-in or attempts to register with an email already in use, the form displays an inline error message explaining the issue. During any API call, network failures trigger a message at the top of the form or page, prompting the user to retry. When loading dashboard data, a failure shows a full-page error banner with a retry button. If the JWT cookie expires while the user is active, any subsequent API call returns an unauthorized response and the app automatically redirects back to the sign-in page.

Should the user lose connectivity in the middle of their session, the UI indicates offline status and disables interactive elements until the connection is restored. Restoring connectivity causes the app to re-fetch data and return to normal operation without losing the user’s place.

## Conclusion and Overall App Journey
From the moment a user lands on the finance dashboard, they are guided to either create an account or sign in. Once authenticated, they arrive at a unified dashboard view layered with global and section-specific navigation. Users interact with summary cards, charts, and tables that draw on static or live financial data. Throughout the application, client-side routing ensures quick transitions, and robust error handling keeps users informed. Logging out returns them cleanly to the sign-in page. This tightly connected flow—from onboarding through daily use—delivers a clear path for users to monitor and analyze their financial information in one place.
